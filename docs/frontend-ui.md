# Frontend UI

Vue + TypeScript + Vite の Web UI は、検索ビューと統計ビューの2画面で構成します。

画面上の表記は `表示項目` に統一します。カード表示では横並びの表カラムとして扱わないため、項目単位の表示切替として説明します。

## Header

ヘッダーは常時表示する情報を最小限にします。

- アプリ名
- 短い説明
- `検索` / `統計` の表示切替

DB件数や取得日時はヘッダーには置かず、検索ビュー内の `data-strip` に表示します。ヘッダーは「何のアプリか」と「どのビューへ移動するか」だけを担います。

## Static Metadata

公開サイトは検索流入を増やさない方針です。`frontend/index.html` には `noindex,nofollow,noarchive` を置き、`frontend/public/robots.txt` は `Disallow: /` とします。OGP、Twitter card、構造化データなどのSNS・検索流入を増やすmetadataは追加しません。

Cloudflare Pages 用の `frontend/public/_headers` には、静的配信向けの基本セキュリティヘッダーを置きます。Worker API 側のヘッダーは `cloudflare/worker/src/index.ts` で返します。

## Search View

検索条件は以下です。

| 条件 | 内容 |
| --- | --- |
| 文字数 | 空欄なら文字数で絞り込まない |
| 作曲者 | 曲詳細由来の `credits.composer` を部分一致検索 |
| 公開年 | 曲詳細ページの年タグ由来の公開年 |
| 根拠タグ | 複数選択可。0件選択ならタグ条件なし |
| 並び順 | 人気度順、文字数昇順/降順、公開年昇順/降順 |

文字数、作曲者、公開年はいずれも任意です。すべて空でも検索できます。

根拠タグの初期状態は `殿堂入り` 以外を選択します。統計ビューから検索へ遷移する場合は、原則として全タグを選択します。ただし、根拠タグの統計項目をクリックした場合は、そのタグ単独で検索します。

検索条件パネルと表示項目メニューの責務:

| UI | 変えるもの | API再検索 |
| --- | --- | --- |
| 検索条件パネル | `length`, `composer`, `year`, `popularity_label`, `sort` | する |
| ページング | `page`, `page_size` | する |
| 表示項目メニュー | カード内に見せる項目 | しない |

状態遷移:

- 初期表示では検索結果を表示しません。ユーザーが検索ボタンを押した後だけ結果を表示します。
- 検索成功後は、デスクトップとモバイルのどちらでも検索条件パネルを閉じます。
- 検索後に根拠タグ、ページ番号、表示件数を変更した場合は再検索します。初回検索前は自動検索しません。
- 統計ビューの項目から検索へ移動する場合は、前回の文字数、作曲者、公開年、並び順、根拠タグ条件をいったんリセットしてから、選択した統計条件だけを適用します。
- 検索ビューと統計ビューを切り替えた場合は、画面上部へスクロールします。

状態の流れ:

```text
初期表示
  -> 検索条件を入力
  -> 検索ボタン
  -> 結果表示
  -> ページング / 表示件数 / 根拠タグ変更で再検索
```

```text
統計ビュー
  -> 統計項目を選択
  -> 前回検索条件をリセット
  -> 選択した条件だけで検索ビューへ移動
  -> 結果表示
```

表示項目メニューは検索条件ではありません。表示項目を変えてもAPI再検索せず、既に取得した結果カードの見せ方だけを変えます。

統計ビューから検索へ移る例:

| クリックした統計 | 適用する検索条件 |
| --- | --- |
| 文字数 `7` | `length=7`、根拠タグは全選択 |
| 公開年 `2021` | `year=2021`、根拠タグは全選択 |
| 根拠タグ `ミリオン達成曲` | `popularity_label=ミリオン達成曲` |
| 作曲者 `ryo` | `composer=ryo`、根拠タグは全選択 |

## Result Cards

検索結果はカード形式で表示します。

- 曲名と Wiki リンクを最初に表示
- 行全体のクリックまたはキーボード操作で詳細を開閉
- 詳細には基本情報、曲紹介、動画、関連動画を表示
- 表示項目メニューで文字数、作曲者、公開年、人気度、根拠タグを追加表示

情報優先順位:

1. 曲名
2. Wikiリンク
3. 詳細アコーディオン
4. ユーザーが選んだ表示項目
5. 補助的な状態表示

表示項目は下部固定バーに置きます。検索結果をスクロール中でも表示項目を変更できるようにするためです。

詳細情報はカードを開いた時点で `/api/song-detail` から取得し、同じページ URL の結果はフロント内で再利用します。詳細APIが `404` を返した場合は、DBに詳細JSONが投入されていない状態として扱います。

詳細アコーディオンの状態:

| 状態 | 表示 |
| --- | --- |
| loading | 読み込み中 |
| success | 基本情報、曲紹介、動画、関連動画 |
| empty | 抽出済みだが該当項目が空 |
| error | 詳細を取得できない旨と再試行可能な状態 |

APIエラーの扱い:

| 種別 | 主な原因 | UI方針 |
| --- | --- | --- |
| `400` | 検索条件やURLがAPI仕様に合っていない | 条件を見直せる短い文言を表示する |
| `404` | 指定した曲詳細がDBにない | 詳細が未投入または見つからない旨を表示する |
| `503` | D1未投入、metadata不足、schema不一致 | DB準備中またはデータ未投入として表示する |
| network error | 通信失敗、Cloudflare経路、ローカルWorker停止 | 再試行できる状態を残す |

エラー文言は、APIの `detail` をそのまま長く見せるより、ユーザーが次にできる行動へ寄せます。開発者向けの切り分けは [web-api.md](web-api.md) と [operations.md](operations.md) を参照します。

## Pagination

検索APIはページングします。Web UI は以下の表示数を選択できます。

- `50`
- `100`
- `200`

下部固定バーには現在の表示範囲、表示数、ページ移動、表示項目メニューを置きます。1ページしかない場合はページャを表示しません。

モバイルでは下部固定バーを圧縮します。

- `先頭` / `最後` は非表示
- `前へ` / `次へ` と現在ページだけを表示
- 表示範囲、表示数、表示項目をコンパクトに配置

## Stats View

統計ビューは `/api/stats` を使います。

表示する統計:

- タイトル文字数ヒストグラム
- 公開年ヒストグラム
- 根拠タグ分布
- 作曲者ランキング

各統計項目をクリックすると、その条件で検索ビューへ移動します。

PCでは統計パネルを一覧表示します。モバイルではカテゴリ切り替えUIを表示し、選択中の統計パネルだけを表示します。左右スワイプでもカテゴリを移動できます。

## Responsive Rules

| 領域 | desktop | mobile |
| --- | --- | --- |
| ヘッダー | 横並びで表示 | 縦に伸びすぎないよう説明文と切替を優先 |
| 検索条件パネル | sticky、検索後は閉じる | sticky、開閉しやすさを優先 |
| 結果カード | 横方向にもカードを並べる | 1列カード中心 |
| 結果操作バー | fixedで下部表示 | 必要項目だけを圧縮して表示 |
| 統計ビュー | 複数パネルを一覧表示 | カテゴリ切替とスワイプ |

主要ラベル:

| 表記 | 用途 |
| --- | --- |
| `検索` | 検索ビュー |
| `統計` | 統計ビュー |
| `検索条件` | 検索パネル |
| `表示項目` | 結果カードに追加表示する項目 |
| `根拠タグ` | 人気度の根拠になったタグ |

画面表記とAPI名:

| 画面表記 | API / 内部名 | 備考 |
| --- | --- | --- |
| 文字数 | `length` | タイトル文字数。空欄なら条件なし |
| 作曲者 | `composer` | 曲詳細由来の作曲者 |
| 公開年 | `year`, `published_year` | 検索条件は `year`、レスポンスは `published_year` |
| 根拠タグ | `popularity_label` | 複数選択可 |
| 並び順 | `sort` | 人気度、文字数、公開年 |
| 表示件数 | `page_size` | `50`, `100`, `200` |

画面上では `表示列` ではなく `表示項目` を使います。結果がカード表示であり、表の列として固定されていないためです。

## Testing Boundary

フロントエンドの自動確認は、軽量な静的検査、production build、Playwright E2E smokeに分けます。

| 確認 | 守るもの | 守らないもの |
| --- | --- | --- |
| `tools/check_frontend_accessibility.py` | focus-visible、主要aria label、キーボード開閉の静的な退行 | 実ブラウザでの見た目、色コントラストの厳密測定 |
| `yarn build` | Vue/TypeScript型、production build成立 | ユーザー操作の成立 |
| `yarn test:e2e` | 検索、詳細、ページング、統計から検索への遷移、mobile幅での主要導線 | 本番D1データの正しさ、細かなレイアウト差分 |

Playwright E2Eは `/api/*` をmockします。目的は、実データの検証ではなく、画面操作が壊れていないことの確認です。本番APIやD1の疎通確認は [testing.md](testing.md#smoke-test) の smoke test で扱います。

現時点ではスクリーンショット差分テストを入れません。環境差で揺れやすく、個人運用の通常開発ループでは保守コストが高いためです。見た目の違和感は、次の観点で手動確認します。

| 手動確認に残す観点 | 理由 |
| --- | --- |
| 余白、密度、視線誘導 | 機械的な合否より人間の見た目判断が必要 |
| sticky検索パネルと下部バーの圧迫感 | 画面サイズや入力状態で印象が変わる |
| 長い曲名、長い作曲者名、長い根拠タグ | 実データの組み合わせで見え方が変わる |
| 統計ヒストグラムの読みやすさ | ラベル分布と件数差で変わる |
